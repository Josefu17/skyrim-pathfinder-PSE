This document outlines the criteria for considering a feature or task complete.

## Code Quality
- All changes adhere to linting (`pylint`) and formatting (`black`) standards.
- The codebase follows team conventions, such as:
  - Snake case (`example_name`).
  - Proper use of docstrings and inline comments for complex logic.

---

## Testing
- Unit tests are written or updated for new features or bug fixes.
- Tests pass locally and in CI.
- Code coverage is maintained or improved:
  - No significant drops in test coverage are allowed.

---

## Functionality
- The feature or task fulfills all defined requirements.
- Edge cases and failure scenarios are handled gracefully.

---

## Documentation
- The README or relevant `.md` files are updated to reflect changes.
- Inline comments are added to clarify non-obvious logic.

---

[back to top](#definition-of-done-dod)
## Collaboration
- Merge requests are reviewed and approved by at least one team member.
- All review feedback is addressed promptly before merging.

---

## Integration
- The CI pipeline passes all stages, including build, test, and deployment.

---

## Approval
- The task is marked as complete by the assigned developer.
- The merge request is formally approved by a reviewer.
